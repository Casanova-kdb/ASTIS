# Phase 2: System Design

## Architecture Overview

ASTIS will follow a layered backend architecture. The first implementation will focus on a Spring Boot backend and REST API, with a frontend added later if needed.

```text
Frontend / API Client
        |
        v
Controller Layer
        |
        v
Service Layer
        |
        +---- AI Service Layer
        |
        v
Repository Layer
        |
        v
Database
```

The AI recommendation module will interact with task, profile, and behaviour data through the service layer. This keeps feature extraction, scoring, and prediction logic separate from basic CRUD operations.

## Planned Backend Layers

### Controller Layer

Responsible for receiving HTTP requests and returning API responses.

### Service Layer

Responsible for business logic and coordination between modules.

### AI Service Layer

Responsible for feature extraction, task scoring, delay risk prediction, and recommendation ranking.

### Repository Layer

Responsible for database access through Spring Data JPA.

## Core Modules

### User Module

Handles user registration, login, authentication, and user profile data for personalisation.

### Task Module

Handles task creation, updates, deletion, status tracking, deadline tracking, and task retrieval.

### Analytics Module

Collects user behaviour data, including task creation, updates, completion, delay, and deletion events.

### AI Recommendation Module

Calculates task priority scores, estimates delay risk, and returns recommended task ordering based on task urgency, user-defined priority, task category, user profile, and historical behaviour.

The AI module is an embedded backend module, not an external text-generation wrapper. The MVP uses deterministic and explainable logic that can later be replaced or extended by a machine learning model.

## Data Flow

1. A user creates or updates a task.
2. The controller receives the request and validates input.
3. The service layer applies business logic.
4. The repository layer stores or retrieves data from the database.
5. The analytics module records the user action.
6. The AI module extracts features from task, profile, and behaviour data.
7. The recommendation module calculates task priority scores and delay risk.
8. The API returns a ranked task list to the user.

## AI Module Design

### Feature Extraction Layer

The feature extraction layer converts stored task and behaviour data into model inputs.

Initial features:

- Task urgency based on deadline proximity
- User-defined task priority
- Historical completion rate
- Task type or category behaviour
- Estimated workload
- Time decay factor
- Preferred study time or time-of-day productivity pattern when available

### Scoring and Prediction Layer

The MVP recommendation engine will use an explainable scoring formula:

```text
score = urgencyScore * w1
      + completionRateScore * w2
      + userPriorityScore * w3
      + timeDecayScore * w4
      + delayRiskScore * w5
```

The delay risk prediction can be implemented as a deterministic risk level in the MVP, such as `LOW`, `MEDIUM`, or `HIGH`, based on deadline proximity, estimated workload, and historical delay patterns.

### AI Output

The recommendation API must return:

- Sorted task list
- Priority score per task
- Delay risk prediction
- Short explanation for each recommendation

This approach is transparent and suitable for an early-stage personal project. A more advanced logistic regression or ranking model can be introduced later if enough behaviour data is collected.
